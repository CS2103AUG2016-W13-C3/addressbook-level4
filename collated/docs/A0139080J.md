                <h3 id="logging">Logging</h3>

                <p> We are using the <code>java.util.logging</code> package for logging. <code>LogsCenter.java</code> is used to manage the logging levels and logging destinations. The <code>Logger</code> for a class can be obtained using <code>LogsCenter.getLogger(Class)</code> which will log messages according to the specified logging level. </p>

                <p>Currently log messages are output through the console and to a <code>.log</code> file.</p>

                <h4>Logging Levels</h4>
                <ul>
                    <li>
                        <code>SEVERE</code>: Critical problems detected which may possibly cause the termination of the application
                    </li>
                    <li>
                        <code>WARNING</code>: Actions that can be continued, but with caution
                    </li>
                    <li>
                        <code>INFO</code>: Information showing the noteworthy actions by the application
                    </li>
                    <li>
                        <code>FINE</code>: Details that are not usually noteworthy but may be useful in debugging e.g. print the actual list instead of just its size
                    </li>
                </ul>

                <h2 id="testing">Testing</h2>

                <p>Tests can be found in the test folder (<code>./src/test/java</code>). We aim to maximise our test coverage, so every class written should be accompanied by a corresponding test class in a similar path within this test folder.</p>

                <h3 id="running-tests">Running Tests</h3>

                <p>Tests can be run in 2 ways:</p>

                <ol>
                    <li>
                        <strong>In Eclipse - </strong> Right-click on the <code>./src/test/java</code> folder and choose <code>Run as</code> &gt; <code>JUnit Test</code>. To run a subset of tests, right-click on a test package, test class, or a test instead.
                    </li>

                    <li>
                        <strong>Using Gradle - </strong> See <a href="UsingGradle.md">UsingGradle.md</a> for how to run tests using Gradle.
                    </li>
                </ol>

                <blockquote>
                        If you are not using a recent Eclipse version (i.e. <em>Neon</em> or later), enable assertions in JUnit tests as described <a href="http://stackoverflow.com/questions/2522897/eclipse-junit-ea-vm-option">here</a>.
                </blockquote>


                <h3 id="types-of-tests">Types of Tests</h3>

                <ol>
                    <li><strong>GUI Tests - </strong>They are System Tests that test the entire application by simulating user actions on the GUI. These are in the <code>guitests</code> package.</li>

                    <blockquote>
                        <strong>Headless GUI Testing</strong> By applying <a href="https://github.com/TestFX/TestFX">TestFX</a> library, GUI tests can be run in the <em>headless</em> mode. In the <em>headless mode</em>, GUI tests do not show up on the screen. That means you can do other things on your device while the tests are running. See <a href="UsingGradle.md#running-tests">UsingGradle.md</a> to learn how to run tests in headless mode.
                    </blockquote>

                    <li>
                        <strong>Non-GUI Tests - </strong>They include:
                        <p>
                            <ol>
                                <li type=i>
                                    <strong>Unit tests:</strong> targeting the lowest-level methods/classes. <br>
                                    E.g. <code>seedu.commando.commons.UrlUtilTest</code>
                                </li>
                                <li type=i>
                                    <strong>Integration tests:</strong> checking the integration of multiple code units (those code units are assumed to be working).<br>
                                    E.g. <code>seedu.commando.storage.StorageManagerTest</code>
                                </li>
                                <li type=i>
                                    <strong>Hybrid tests:</strong> testing multiple code units as well as how the are connected together.<br>
                                    E.g. <code>seedu.commando.logic.LogicManagerTest</code>
                                </li>
                            </ol>
                        </p>
                    </li>
                </ol>

                <h3 id="add">
                    Adding To-Dos: <code>add</code>

                <p>The very first thing you'll want to do is to add a to-do. Each to-do contains minimally a description. In addition, it can be optionally associated with multiple tags and a time constraint. Depending on nature of the time constraint, to-dos are categorised into 2 types: <em>Events</em> (shown in the left panel) and <em>Tasks</em> (shown in the right panel):</p>

                    <img src="https://github.com/CS2103AUG2016-W13-C3/main/blob/master/docs/images/Ui.png?raw=true" width="600">

                <h4>Adding Events</h4>

                <p>An <em>Event</em> is an activity happening within a time window, with <code>start_datetime</code> and <code>end_datetime</code> fields to represent the time window. To associate tags with the <em>Event</em>, prefix each of them with a <code>#</code>. In <em>CommanDo</em>, every <em>Event</em> from the current day onwards is listed in the left panel chronologically:</p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> add <span>&lt;description&gt;</span> from <span>&lt;start_datetime&gt;</span> to <span>&lt;end_datetime&gt;</span> [daily/weekly/monthly/yearly] [#<span>&lt;tag&gt;</span>...]</pre>

                <pre><span class="inputstart">&gt;&gt;</span> add <span>meeting with Dr Tan</span> from <span>next Saturday 9am</span> to <span>10am </span>#<span>work </span>#<span>important</span></pre>

                <h3 id="edit">Editing To-Dos: <code>edit</code></h3>

                <p>More often than not, you'll need to edit the fields of a to-do after adding it, such as when you made a mistake, when an <em>Event</em> gets postponed, or when you receive more details for a <em>Task</em>.</p>

                <h4>Editing Events</h4>

                <p>By including suitable fields and their keywords, you'll be able to replace the description, the time window and the tags of an <em>Event</em>. Note that if tags are provided in an edit, the old tags will be completely replaced by the new ones. To remove the time window or the tags of a <em>Event</em>, use the <a href="#delete"><code>delete</code> command</a> instead.</p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> edit <span>&lt;index&gt;</span> [<span>&lt;description&gt;</span>] [from <span>&lt;start_datetime&gt;</span> to <span>&lt;end_datetime&gt;</span> [daily/weekly/monthly/yearly]] [#<span>&lt;tag&gt;</span>...]</pre>

                <span><strong>Examples:</strong></span> <span>(given an <em>Event</em> at <em>Index</em> 1)</span>
                <pre><span class="inputstart">&gt;&gt;</span> edit <span>1 #projectABC</span></pre>
                <span>What it does:  Adds or replaces the tags of that event with the tag <code>projectABC</code>.</span>
                <pre><span class="inputstart">&gt;&gt;</span> edit <span>1</span> from <span>today 1400</span> to <span>today 1600 #important #work</span></pre>
                <span>What it does:  Changes the time window of that event and replaces its tags by <code>important</code> and <code>work</code>.</span>

                <h4>Editing Tasks</h4>

                <p>By including suitable fields and their keywords, you'll similarly be able to replace the description, the deadline and the tags of a <em>Task</em>. To remove the deadline or the tags of a <em>Task</em> instead, use the <a href="#delete"><code>delete</code> command</a>.</p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> edit <span>&lt;index&gt;</span> [<span>&lt;description&gt;</span>] [by <span>&lt;due_datetime&gt;</span>] [#<span>&lt;tag&gt;</span>...]</pre>

                <span><strong>Examples:</strong></span> <span>(given a <em>Task</em> at <em>Index</em> 2)</span>
                <pre><span class="inputstart">&gt;&gt;</span> edit <span>2 shopping trip to IKEA</span></pre>
                <span>What it does:  Changes the description of that task to <code>shopping trip to IKEA</code>.</span>
                <pre><span class="inputstart">&gt;&gt;</span> edit <span>2</span> by <span>6 feb</span></pre>
                <span>What it does:  Adds or changes the deadline of that task.</span>

                <h3 id="export">Exporting: <code>export</code></h3>

                <p>If you want to copy your to-do list data file in your current save location to another local destination, there's no need to manually do so. Instead, use the <code>export</code> command to save a copy of the current to-do list data to the specified file at given file path.</p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> export <span>&lt;file_path&gt;</span></pre>

                <h3 id="import">Importing: <code>import</code></h3>

                <p>If you already have an existing to-do list data file (possibly from your other devices) and want to use it in <em>CommanDo</em>, the <code>import</code> command can import a valid data file. <strong>Note that it overrides the current to-do list with the imported to-do list!</strong></p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> import <span>&lt;file_path&gt;</span></pre>

                <h3 id="undo">Undoing: <code>undo</code></h3>

                <p>Scenario: you deleted a to-do by accident and you want to undo your last modification. With the <code>undo</code> command, the most recent <code>add</code>, <code>edit</code>, <code>delete</code>, <code>clear</code> and <code>import</code> would be reverted. Multiple <code>undo</code> commands can be used in succession, as long as there is something to undo. <strong>Note that you can only undo commands in your current session, which ends when you close the application.</strong></p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> undo</pre>

                <h3 id="redo">Redoing: <code>redo</code></h3>

                <p>Scenario: You deleted a to-do by accident. You had used <code>undo</code> to restore that to-do, but eventually realised that the to-do isn't worth keeping after all. The <code>redo</code> command reverts the most recent <code>undo</code>, and can be used in sucession. <strong>What you can redo is also limited to the current session.</strong></p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> redo</pre>

                <h3 id="clear">Clearing To-Do List: <code>clear</code></h3>

                <p>If you want to reset your to-do list, use the <code>clear</code> command. <strong>ALL to-dos, including those finished and archived, will be removed.</strong> Use the <code>undo</code> command if you cleared your to-do list by accident.</p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> clear</pre>

                <h3 id="exit">Exiting: <code>exit</code></h3>

                <p>Finally, you are done with managing your to-do list and want to close the program. You can either click on the close button or use the <code>exit</code> command provided in <em>CommanDo</em>. This command closes the application, and all the changes to your to-do list are saved.</p>

                <span><strong>Command Format:</strong></span>
                <pre class="command-format"><span class="inputstart">&gt;&gt;</span> exit</pre>

                <h3>2. How do I save my to-do list after modifying it?</h3>

                <p>Your to-do list is automatically saved to your file system after any command that changes the data. There is no need to save manually.</p>
